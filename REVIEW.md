# Yorimichi プロフェッショナルレビュー

**レビュー日**: 2026-03-16
**対象バージョン**: 1.0.10
**レビュアー**: ソフトウェアエンジニア（モバイルアプリ専門）

---

## 総合評価

Yorimichiは「位置情報×買い物メモ」というニッチながら実用的なコンセプトのアプリです。ネイティブGeofencingClientの採用、Zustand + MMKVによる高速な状態管理、Firestoreによるリアルタイム共有など、技術的基盤は堅実です。以下、プロの観点から多角的にレビューします。

---

## 1. 収益化（Monetization）

### 現状の評価
- **Freemium + 広告 + サブスク + クーポン + トライアル** の5層構造は多彩
- RevenueCat統合により購入管理は堅牢

### 改善提案

#### 1-1. 無料プランの制限値が厳しすぎる
- **現状**: メモ3件、場所2件/メモ、アイテム10件/メモ
- **問題**: 無料ユーザーがアプリの価値を十分に体感する前に壁にぶつかる。買い物リストで10アイテムは日常使いに足りない
- **提案**: メモ5件、場所3件/メモ、アイテム20件/メモ に緩和。ユーザーに「これは便利だ」と感じさせてから課金を促す方が、コンバージョン率が上がる

#### 1-2. トライアルの自動開始と購入導線
- **現状**: トライアルはPremiumScreen上のボタンで手動開始（7日間）
- **提案**:
  - インストール後3日目に「プレミアム体験しませんか？」のプッシュ通知で誘導
  - トライアル残り1日でアプリ内バナーを表示し、自然なアップグレード導線を作る
  - トライアル終了後24時間以内に「特別割引」オファーを出す（RevenueCatのプロモオファー機能）

#### 1-3. インタースティシャル広告のタイミング
- **現状**: Settings画面遷移時に表示（メモ5件以上の無料ユーザー）
- **問題**: Settings画面は「設定したい」という明確な目的がある遷移。ここで広告を出すとフラストレーションが高い
- **提案**: 代わりにメモ完了時（全アイテムチェック時）に表示。「タスク達成」のポジティブな瞬間なのでネガティブ印象が薄い

#### 1-4. 買い切りオプションの追加検討
- 日本市場ではサブスク疲れが進んでいる。「ライフタイムプラン」（例: 年額×3程度）を追加すると、サブスクに抵抗があるユーザー層を取り込める
- RevenueCatは非消費型IAPにも対応している

#### 1-5. クーポンの活用幅を広げる
- **現状**: クーポンは30日間のプレミアム解放
- **提案**: インフルエンサー/ブロガー向けに発行して口コミを促進。紹介コード制度（既存ユーザーが友達にコードを共有→両者に7日間プレミアム）で有機的な成長を狙う

## 2. UX/UI（ユーザー体験）

### 2-1. オンボーディング改善
- **現状**: OnboardingScreenは一度だけ表示
- **問題**: ジオフェンスという概念は一般ユーザーには馴染みが薄い。一回のチュートリアルだけでは価値が伝わりにくい
- **提案**:
  - 初回のメモ作成時に「場所を登録すると、近くに来た時に通知でお知らせします」とコーチマーク表示
  - 初回のジオフェンス通知受信時に「使い方のコツ」ポップアップで成功体験を強化

### 2-2. メモ作成フローの最適化
- **現状**: `handleAddNewMemo` → LocationPickerScreen に直接遷移
- **問題**: 新規メモ作成時にまず場所選択から始まるのは直感的でない。「何を買うか」が先にあるのが自然
- **提案**:
  1. 「+」ボタン → メモタイトル入力 → アイテム追加 → 場所追加（任意）の順序に
  2. 場所なしのメモも作れるようにする（純粋な買い物リストとしても使えるように）
  3. これにより「タスク管理アプリ」としてのユースケースも広がる

### 2-3. 検索・フィルタ機能の追加
- **現状**: MemoListScreenにはメモの一覧表示のみ
- **提案**:
  - メモのタイトル検索
  - 場所ベースのフィルタ（「スーパーA に関連するメモだけ表示」）
  - 完了/未完了フィルタ
  - ソート機能（作成日/更新日/期限日/場所距離順）
  - メモ数が増えると（特にプレミアムユーザーは1000件まで可能）、検索なしでは使い物にならなくなる

### 2-4. 削除時のUndoの不具合
- `MemoListScreen.tsx:132-143` で `handleDelete` はAlert確認後に `deleteMemo` を呼ぶが、Snackbar用の `deletedMemo`/`deleteSnackbarVisible` を一度もセットしていない
- Snackbar + Undo のコードは存在するが（L303-316）、実際には呼ばれない = デッドコード
- **修正提案**: Alert確認を廃止し、即座に削除 + Snackbar表示でUndo可能にする。これはGmail風の「先に実行、後から取り消し」パターンで、操作のスピード感が上がる

### 2-5. アクセシビリティ
- `accessibilityLabel` / `accessibilityRole` が殆どのコンポーネントに設定されていない
- 視覚障害ユーザーがスクリーンリーダーで操作できない状態
- **提案**: 主要なタッチ要素（ボタン、チェックボックス、カード）に最低限のアクセシビリティ属性を追加

### 2-6. ダークモード対応
- **現状**: ライトモードのみ（カラーコードがすべてハードコード）
- **提案**: `src/theme/colors.ts` を活用してダークテーマを追加。OS設定に連動させる。特に夜間の買い物（退勤後のスーパー）で使うユースケースが多いアプリなので効果的

## 3. アーキテクチャ/コード品質

### 3-1. 画面コンポーネントの肥大化
- `MemoDetailScreen.tsx`（915行）に状態管理、ビジネスロジック、UI描画がすべて詰め込まれている
- **提案**: カスタムフックへの切り出し
  - `useSharedMemoSync(memoId)` — Firestore同期ロジック
  - `useMemoActions(memoId)` — チェック/アンチェック/共有等のハンドラ
  - `useNotifyCooldown(shareId)` — 通知クールダウン管理

### 3-2. StyleSheetの分離
- 各画面ファイルの半分近くがStyleSheet定義（MemoDetailScreen: 200行以上）
- **提案**: `src/styles/` ディレクトリに画面ごとのスタイルファイルを分離。さらに共通カラー/スペーシング定数を `theme` に統合

### 3-3. 型安全性の向上
- `t(('premium.' + row.label) as any)` のような `as any` キャストが散見される（PremiumScreen.tsx:250）
- i18nのキーを型安全にするため、`resources` の型定義を活用してキー補完を有効にすべき

### 3-4. エラーハンドリングの統一
- 各画面で `try/catch` + `Alert.alert` のパターンが繰り返されている
- **提案**: 共通の `useErrorHandler` フックまたはグローバルエラーバウンダリで一元管理

### 3-5. マジックナンバーの定数化
- `3000`（ポーリング間隔）、`800`（デバウンス遅延）、`5000`（Snackbar表示時間）等がコード中に散在
- `src/config/constants.ts` に集約すべき

## 4. パフォーマンス

### 4-1. FlatListの最適化不足
- `MemoListScreen.tsx:254` のFlatListに `getItemLayout` が未設定。メモ数が多い場合にスクロールパフォーマンスが低下する
- `renderItem` 内で毎回 `filter` を実行している（L147: `item.items.filter(i => !i.isChecked).length`）
- **提案**: `useMemo` でフィルタ済み値をキャッシュし、`getItemLayout` を追加

### 4-2. 共有メモの同期負荷
- `MemoListScreen.tsx:49-73` で画面フォーカスの度に全共有メモを一括取得する `syncAllSharedMemos` が走る
- メモ数が増えると通信量・Firestore読み取り数が増大（コスト直結）
- **提案**:
  - `updatedAt` のタイムスタンプ比較で差分のみ同期
  - バックグラウンドでの定期同期に切り替え（例: 5分おき）

### 4-3. Zustandのsubscribe + syncWidget
- `memoStore.ts:528-530` でメモの**全変更**に対してウィジェット同期が走る
- チェックボックスのトグルのような高頻度操作でも毎回
- **提案**: `debounce` でウィジェット同期を間引く（500ms程度）

## 5. セキュリティ

### 5-1. Firestoreルールの改善
- `firestore.rules` にレガシードキュメントへのフォールバック期限（2026-06-30）が設定されているのは良い対応
- ただし `collaborators` 配列にデバイスIDを使用しているのはセキュリティ上弱い。デバイスIDは推測・偽装可能
- **提案**: Firebase Auth UIDベースの認証に完全移行し、レガシーフォールバックの期限を早める

### 5-2. デバイスID依存
- `getDeviceId()` がバックアップ・クーポン・共有の識別子として広く使われている
- デバイスリセットや機種変更でデータが失われるリスクがある
- **提案**: Firebase Auth UID を主キーとし、デバイスIDはセカンダリにする移行計画を策定

### 5-3. 匿名認証の脆弱性
- 匿名認証はアカウント紐付けができない。アプリ削除→再インストールで新しいUIDが発行される
- **提案**: 将来的にメールリンク認証やApple/Googleサインインを追加し、データのポータビリティを確保

## 6. テスト

### 6-1. テストカバレッジの拡充
- **現状**: 13のJestテストスイートはサービス層を中心にカバー
- **不足点**:
  - 画面コンポーネントのテストがほぼない（App.test.tsx以外）
  - ジオフェンス通知のE2Eテストが未実装
  - エッジケース（オフライン時の動作、同時編集の競合解決）のテストがない
- **提案**: React Native Testing Libraryで主要画面のインテグレーションテストを追加

### 6-2. Maestro E2Eの拡充
- smokeテストは基本フローをカバーしているが、プレミアム機能・共有機能・バッジ解除のフローが未実装
- **提案**: CI/CDパイプラインでMaestro E2Eを自動実行する仕組みを構築

## 7. 成長戦略・競争力強化

### 7-1. カテゴリ特化テンプレート
- 「スーパー」「ドラッグストア」「コンビニ」等の店舗カテゴリに応じた買い物リストテンプレートを提供
- よく買うアイテムをワンタップで追加できる「クイック追加」機能

### 7-2. AIアシスト機能（差別化要素）
- 買い物履歴から「今週買い忘れていそうなもの」をサジェスト
- 写真からレシピの材料を自動抽出してメモ化
- これらはプレミアム限定機能にすることで課金動機にもなる

### 7-3. Apple Watch / Wear OS 対応
- 店舗到着時にウォッチで買い物リストを確認→チェックの流れは非常に自然
- ジオフェンス通知との組み合わせで「手元で完結」する体験を提供

### 7-4. ウィジェットの強化
- **現状**: Androidウィジェットは基本的なメモ表示
- **提案**: ウィジェットから直接アイテムのチェック/アンチェックが可能に。ホーム画面だけで買い物完了できる体験

### 7-5. 店舗チェーンとの連携
- 特定のスーパーの特売情報と連携し、「今日安い商品がメモにあります」通知
- 将来的にアフィリエイトモデルとしての収益化も可能

### 7-6. iOS対応の本格化
- **現状**: iOS用のプロジェクトは存在するが、ネイティブモジュール（GeofenceModule）はAndroid専用
- iOS市場は課金率が高いため、CoreLocation Geofencingの実装は収益面で重要

## まとめ

Yorimichiは技術的に堅実なアプリですが、**ユーザーが価値を感じる前に制限にぶつかる**という収益化の課題があります。まず無料プランの制限緩和とメモ作成フローの最適化で「使い続けたい」と思わせてから課金に導くことが最重要です。

コードベースは単一画面への責務集中が進んでおり、今後の機能追加に備えてカスタムフックへの分離を進めるべきです。iOS市場への本格展開は収益を大きく伸ばすポテンシャルがあります。
